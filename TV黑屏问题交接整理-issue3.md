# TV 黑屏问题交接整理（issue #3）

## 一、问题概述

### 1.1 问题链接
- `GitHub Issue`：<https://github.com/June6699/dart_simple_live/issues/3>
- 标题：`[BUG] 画面异常`

### 1.2 当前核心现象
- `Android TV` 端在部分真机上进入直播间后出现“`有声音，但没有视频画面`”。
- 该问题不是单个平台单个房间的问题，`虎牙 / 斗鱼 / 抖音 / B站` 都会出现。
- 用户最新反馈是：之前的“`花屏`”现象已经没有了，但仍然是“`黑屏有声`”。

### 1.3 当前不能再继续假设的方向
- 不是某一个房间导致。
- 不是某一个直播平台导致。
- 不是用户下载错了 `x86_64` 包或其他明显错误包。
- 目前还不能说“问题已经解决”，因为真实设备反馈明确仍未解决。

## 二、用户设备与环境信息

### 2.1 设备信息
- 设备：`NVIDIA SHIELD Android TV`
- 系统：定制版 `Android TV 11`
- 架构：`arm64-v8a`
- 分辨率：`3840x2160`
- 设备特点：高性能安卓电视盒子，系统原生偏 `Android TV` 规范，可能存在特定 `MediaCodec / EGL / Surface` 渲染链路差异。

### 2.2 用户测试结果
- 旧版 `TV 1.4.3`：用户反馈可以正常使用。
- `1.4.3` 之后的版本：用户反馈一直有问题。
- 当前测试包：`SimpleLive-TV-arm64-v8a-release.apk`
- 最新反馈：
  - `四个平台所有黑屏有声`
  - `解码开关和兼容模式都已经测过`
  - 之前的“花屏”现象已经消失，但“黑屏有声”仍在。

## 三、问题背景与排查过程

### 3.1 最初背景
- 之前没有本地 `TV` 真机设备，只能靠代码分析和用户反馈判断。
- 后来在本地启动了 `Android Studio` 模拟器进行复现与测试。
- 本地模拟器路径中，曾观察到播放器相关的 `EGL / GL / gpu` 输出异常，因此一度怀疑是视频渲染链路问题。

### 3.2 本地测试环境
- 模拟器：`Android Emulator - Medium_Phone`
- 系统：`Android 16 / API 36 / x86_64`
- 本地模拟器测试结果：虎牙、斗鱼、抖音可正常出画面。

### 3.3 已做过的操作
- 整理并重新发布了 `tv_v1.7.0` 的 `TV release` 资产。
- 删除了重复命名的旧 `apk` 资产，只保留：
  - `SimpleLive-TV-arm64-v8a-release.apk`
  - `SimpleLive-TV-armeabi-v7a-release.apk`
  - `SimpleLive-TV-x86_64-release.apk`
- 更新了 `README.md`、`UPDATE.md`、`assets/tv_app_version.json`、`release` 说明。
- 在 `TV` 端设置说明里补充了黑屏排查提示。
- 把 `TV` 播放相关依赖和路径调整到当前这版可在本地模拟器出画面的状态。

### 3.4 当前代码层面与版本层面的已知事实
- 当前 `TV` 端使用的 `media_kit / media_kit_video / media_kit_libs_video` 版本，并不比 `TV 1.4.3` 更新。
- 也就是说，这次问题不一定是“单纯升级 `media_kit` 导致”。
- 但是当前版本与 `TV 1.4.3` 相比，整体工程、播放器周边、渲染链路、Flutter 运行时环境都可能已有组合差异。

## 四、目前更像的原因

### 4.1 更可能的方向
- 更像是 `真机视频渲染链路兼容性问题`，不是流解析问题。
- 因为现象是：
  - 音频正常
  - 直播流大概率拿到了
  - 房间解析大概率正常
  - 但视频层没有正确渲染出来

### 4.2 重点怀疑方向
- `arm64-v8a` 真机路径与 `x86_64` 模拟器路径存在差异。
- `NVIDIA SHIELD Android TV / Android 11` 上的 `MediaCodec / Surface / EGL / GPU` 输出链路存在兼容性问题。
- `4K / HDR / Dolby Vision` 一类输出环境，也可能会放大这个兼容性问题。
- 问题可能不是“能不能解码”，而是“解出来之后能不能正确显示到屏幕上”。

### 4.3 当前不能下定论的点
- 不能证明现在这个问题只在 `arm64-v8a` 上出现，因为用户还没测 `armeabi-v7a` 包。
- 不能证明一定是 `Flutter 3.41.9` 单独导致，但它值得继续怀疑。
- 不能证明一定是 `media_kit` 单独导致，因为 `1.4.3` 用的版本并不比当前旧很多，甚至核心版本看起来一致。

## 六、建议下一会话优先做的事

### 6.1 先补的信息
- 让用户测试 `SimpleLive-TV-armeabi-v7a-release.apk`
  - 目的不是让他长期换包
  - 而是判断是不是 `arm64-v8a` 路径特有问题
- 让用户提供进房后约 `10` 秒的 `adb logcat`
  - 重点看：`MediaCodec`、`EGL`、`Surface`、`mpv`、`media_kit`

### 6.2 更适合新会话继续查的方向
- 对比 `TV 1.4.3` 与当前 `TV 1.7.0`：
  - `pubspec.yaml`
  - `pubspec.lock`
  - 播放器初始化逻辑
  - `AndroidManifest.xml`
  - `Flutter` 版本和构建链
- 看看是否需要专门提供一个“`SHIELD 兼容测试包`”，而不是泛泛增加很多 `apk`。
- 如果要提供额外 `apk`，应当是“有明确测试目的”的版本，而不是一口气放很多让用户盲试。

## 七、当前最重要的结论

### 7.1 一句话结论
- 这个问题目前更像是：`SHIELD 真机的视频渲染兼容性问题`，不是房间问题，也不是平台问题。

### 7.2 当前最合理的下一步
- 不要继续假装“已经修好”。
- 先围绕 `arm64-v8a / armeabi-v7a` 对比、`adb logcat`、以及 `1.4.3` 到当前版本的播放器链路差异继续查。
